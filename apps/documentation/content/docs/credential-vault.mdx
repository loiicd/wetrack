## What is the Credential Vault?

The Credential Vault lets you reference external API credentials in your DataSource configurations **without ever putting secret values in your code or repository**.

It works like `.env` files — but for dashboards:

- **In your code:** Reference a credential by its label (a string)
- **In the WeTrack UI:** Set the actual secret value, encrypted at rest
- **At data fetch time:** WeTrack automatically injects the credential as an HTTP header

## Concept

```typescript
// ✅ What you write in code — label only, no secret
new DataSource("my-api", {
  type: "rest",
  config: {
    url: "https://api.example.com/data",
    method: "get",
    credential: "my-api-key",  // label, not the actual key
  },
})

// ❌ What you NEVER do
new DataSource("my-api", {
  type: "rest",
  config: {
    url: "https://api.example.com/data",
    method: "get",
    headers: { "Authorization": "Bearer sk-1234..." },  // never in code!
  },
})
```

## Setting up a credential

1. Go to **Settings → Credentials** in the WeTrack app
2. Click **Add Credential**
3. Choose a label (e.g. `my-api-key`) and type
4. Enter the secret value
5. Save

The value is encrypted with AES-256-GCM before storage. WeTrack never exposes the raw value.

## Credential types

| Type | HTTP Header | Use case |
|------|------------|----------|
| `bearer` | `Authorization: Bearer <value>` | OAuth 2.0, JWT tokens |
| `api-key` | `X-Api-Key: <value>` | API key authentication |
| `basic` | `Authorization: Basic <base64(value)>` | HTTP Basic auth (`user:password`) |
| `header` | `<headerName>: <value>` | Custom header (configure `headerName`) |

## Example: Bearer token

**In the UI:**
- Label: `github-token`
- Type: `bearer`
- Value: `ghp_xxxxxxxxxxxxx`

**In your stack:**

```typescript
new DataSource("github-api", {
  type: "rest",
  config: {
    url: "https://api.github.com/repos/myorg/myrepo/issues",
    method: "get",
    credential: "github-token",
  },
})
```

WeTrack will automatically inject `Authorization: Bearer ghp_xxxxxxxxxxxxx` when fetching data.

## Security

- Values are encrypted with **AES-256-GCM** using a PBKDF2-derived key
- Credentials are scoped to your organization — other orgs cannot access them
- The encryption key (`VAULT_SECRET`) is never stored in the database
- Raw values are never returned via any API endpoint
